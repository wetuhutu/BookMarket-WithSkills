# Project Logger - 项目日志与周志生成器

这个skill通过扫描项目完整文件和docs目录下的文档，自动生成项目开发流程的日志和周志。

## 何时使用

当用户需要以下内容时调用此skill：
- 生成项目开发日志
- 创建项目周志报告
- 记录项目开发进度
- 归档项目开发历程
- 生成项目开发流程文档

## 核心功能

### 1. 项目分析

- **完整文件扫描**：扫描整个项目目录结构
- **文档分析**：读取并分析docs目录下的所有文档
- **代码分析**：分析代码文件，理解项目功能
- **技术栈识别**：识别项目使用的技术和框架
- **功能模块识别**：识别项目的核心功能模块

### 2. 日志生成

生成4篇日志，分为2周，每周2篇日志：

#### 第一周日志
- **日志1**：项目初始化和架构设计
- **日志2**：核心功能开发

#### 第二周日志
- **日志3**：功能完善和测试
- **日志4**：部署和优化

### 3. 周志生成

生成2篇周志，每篇周志包含对应的2篇日志：

#### 第一周周志
- 包含日志1和日志2
- 总结第一周工作内容
- 记录遇到的问题和解决方案

#### 第二周周志
- 包含日志3和日志4
- 总结第二周工作内容
- 记录项目完成情况

## 执行步骤

### Phase 1: 项目扫描与分析

1. **扫描项目结构**
   - 识别项目类型（前端、后端、全栈）
   - 列出所有源代码文件
   - 识别配置文件
   - 识别文档文件

2. **分析项目文档**
   - 读取 `docs/` 目录下的所有Markdown文档
   - 提取项目概述、架构设计、技术栈等信息
   - 分析API文档、数据库设计等
   - 理解项目目标和范围

3. **分析代码文件**
   - 扫描后端代码（Java、Python等）
   - 扫描前端代码（Vue、React等）
   - 识别核心功能模块
   - 理解业务逻辑

4. **识别技术栈**
   - 从package.json、pom.xml等文件识别依赖
   - 识别框架和库
   - 识别数据库和中间件

### Phase 2: 日志生成

- 项目背景和目标


```markdown

2026-02-01 至 2026-02-07

## 本周工作总结

### 完成的工作
1. 项目初始化和架构设计
   - 搭建项目基础框架
   - 设计系统架构
   - 设计数据库结构
   - 设计API接口

2. 核心功能开发
   - 实现用户认证模块
   - 实现文件管理模块
   - 实现JWT认证

- 使用JWT无状态认证

## 本周日志
- [日志1 - 项目初始化和架构设计](./logs/day1.md)
- [日志2 - 核心功能开发](./logs/day2.md)

- 前端页面开发

- 功能测试
```

## 日志内容要求

### 每篇日志必须包含

1. **日期**：日志记录的日期
2. **工作内容**：详细记录当天完成的工作
3. **技术要点**：记录使用的技术和方法
4. **遇到的问题**：记录开发中遇到的问题
5. **解决方案**：记录问题的解决方案
6. **下一步计划**：记录后续工作计划

### 每篇周志必须包含

1. **时间范围**：周志覆盖的时间段
2. **工作总结**：总结本周完成的工作
3. **技术亮点**：记录本周的技术亮点
4. **遇到的问题**：记录本周遇到的主要问题
5. **日志索引**：列出本周包含的日志
6. **下周计划**：规划下周的工作内容

## 分析要点

### 从项目文件中提取

1. **项目类型**：识别是Web应用、移动应用还是其他类型
2. **技术栈**：识别使用的编程语言、框架、数据库
3. **功能模块**：识别项目的核心功能模块
4. **架构模式**：识别使用的架构模式（MVC、微服务等）

### 从文档中提取

1. **项目概述**：从01-project-overview.md提取项目背景和目标
2. **架构设计**：从02-architecture-design.md提取架构信息
3. **技术栈**：从03-technology-stack.md提取技术信息
4. **数据库设计**：从04-database-design.md提取数据库信息
5. **API文档**：从05-api-documentation.md提取API信息
6. **开发规范**：从06-development-standards.md提取开发规范
7. **测试指南**：从07-testing-guide.md提取测试信息
8. **部署指南**：从08-deployment-guide.md提取部署信息
9. **运维手册**：从09-operations-manual.md提取运维信息

## 日志生成原则

1. **真实性**：基于实际项目情况生成日志
2. **完整性**：覆盖项目开发的完整流程
3. **逻辑性**：日志内容要符合开发逻辑
4. **专业性**：使用专业的技术术语
5. **可读性**：日志内容要清晰易懂

## 特殊情况处理

### 如果项目文档不完整

1. **缺少项目概述**：从代码中推断项目功能
2. **缺少架构文档**：从代码结构推断架构
3. **缺少API文档**：从Controller代码推断API
4. **缺少数据库文档**：从Mapper文件推断数据库

### 如果项目规模很小

1. **调整日志内容**：适当简化日志内容
2. **合并工作内容**：将相关工作合并记录
3. **增加细节**：补充更多技术细节

### 如果项目规模很大

1. **突出重点**：只记录核心功能开发
2. **分模块记录**：按模块组织日志内容
3. **增加日志数量**：可以适当增加日志数量

## 最佳实践

1. **先分析后生成**：充分分析项目后再生成日志
2. **引用文档**：在日志中引用相关文档
3. **添加代码示例**：在日志中添加关键代码示例
4. **记录决策**：记录重要的技术决策
5. **总结经验**：在周志中总结经验教训
